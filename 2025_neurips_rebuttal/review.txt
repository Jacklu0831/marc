# Reviewer 1

Summary:
The paper proposes Context Tuning (CT), a lightweight, inference-time adaptation technique that directly optimizes the context fed to a frozen large language model instead of its parameters. Two realizations are introduced: CT-Prompt, which learns a soft prompt initialized with the actual few-shot demonstration tokens, and the more efficient CT-KV, which initializes and tunes layer-wise key-value prefixes derived from those demonstrations. Both variants incorporate Leave-One-Out masking to prevent label leakage and Token Dropout to improve generalization. Across four benchmarks, CT outperforms standard in-context learning, Prompt/Prefix Tuning, and matches or exceeds the TTT approach while using roughly half the training time. Further, CT-KV can stack on top of TTT for additional gains.

Strengths And Weaknesses:
Strengths:

CT-KV replaces the quadratic cost of re-encoding soft prompts with a linear-time ke-value prefix update, cutting per-task training time by roughly 40-60% relative to CT-Prompt and about half relative to Test-Time Training (TTT) while matching or slightly exceeding their accuracy.

Context Tuning can be stacked on top of model-centric methods like TTT for additional gains; its demonstration-initialized prefixes also reduce seed variance and avoid label leakage via Leave-One-Out masking and Token Dropout. These qualities make CT easy to plug into existing pipelines

Weaknesses:

CT-KV stores a full key-value prefix per layer per demonstration token. For long contexts or deeper models this could inflate GPU memory, yet the paper does not profile memory usage or compare it with alternatives like LoRA or compressed-cache methods.

The paper fixes the number of demonstrations per benchmark without systematically varying k within any dataset. I strongly recommend the authors include an ablation over the number of few-shot examples—even on one or two representative tasks—to demonstrate how Context Tuning‘s performance and robustness depend on demonstration size.

We appreciate the reviewer’s request for deeper theoretical insight. While Context Tuning was developed as a high-performing and practical method for LLM few-shot learning, we also provide a simple argument below to help explain why it outperforms ICL.

As discussed in Section 5.7, ICL constructs a task-specific KV-cache
 from the demonstration pairs, which is then used to solve the query input. CT-KV extracts the same KV-cache but directly optimizes it via gradient descent on the demonstration pairs before using it to solve the query input. Assuming the demonstration pairs are sampled i.i.d. from the same underlying task distribution as the query, minimizing the loss on the demonstration pairs can improve the cache’s ability to generalize to the held-out query. This is analogous to fine-tuning model weights on a training set to improve performance on a held-out test set. We will include a more thorough explanation of why Context Tuning improves upon our baselines in the paper to enhance its clarity and theoretical depth.

# Final Decision by Area Chair

Decision: Reject
Comment:
Contributions: This paper investigates gradient-based tuning of soft prompts that will encourage a pretrained Transformer to better solve a mapping task x ↦ y on training examples (x,y).

The soft prompt is initialized to a hard prompt.

In prior work on prompt tuning or prefix tuning (called "CT-Prompt" here), the layer-0 embeddings of the prompt tokens are tuned, and the upper-layer embeddings are constructed from them in the usual way, using the pretrained parameters. After each parameter update, the upper-layer embeddings and their KV vectors must be recomputed using self-attention, which is slow.
The paper advocates a modification (called "CT-KV") in which the prompt's KV cache is tuned at all layers simultaneously (all KV vectors are detached from their ancestors in the computation graph). This is claimed to result in more accurate models. It is also claimed to speed up training because it doesn't need to use self-attention to recompute the upper layers.
The paper focuses specifically on the case where the soft prompt is initialized to a sequence of few-shot examples, x₁y₁x₂y₂...xₖyₖ. There is nothing technically special about this case, except that they reuse the same (xᵢ,yᵢ) examples as the training examples for tuning. This requires a leave-one-out mechanism where they are masked in the prompt. As a result, there are k different prompts. The same prompt isn't used across training examples, reducing the opportunity for shared work and increasing the practical importance of the speedup above.

Strengths: The idea is reasonable, the experiments are thorough, and the method appears to be effective.

One way of interpreting the results is this:

ICL is not as good as gradient-based tuning like Prompt Tuning, Prefix Tuning, and PEFT methods, but if you combine them ("Context Tuning" = "CT") then you do best of all.
CT-KV is both more efficient and more accurate than CT-Prompt.
Weaknesses: The core CT-KV contribution is incremental engineering: directly tuning the KV cache of the prompt. Sensible, but a small contribution. (The method is rather like Prefix Tuning but with a long, well-initialized prefix.)

The CT-KV contribution appears to be applicable beyond ICL, but it is unfortunately only evaluated for ICL. For example, if prompting with verbal instructions (for an instruction-tuned model), one could tune the KV cache of that prompt to do well on the training examples (x,y).

Some reviewers appear to be skeptical of ICL, because the long prompts become increasingly expensive at both training and test time as the number
 of training examples grows. (While the rebuttal did try varying
, it only reported accuracy, not also training time as in the paper.)

The idea of combining ICL and gradient-based tuning is worthwhile as well, but it's still a little hard to discern what's going on. It would be helpful to vary the following dimensions in a factorial design, evaluating accuracy and computational cost for each combination of settings:

no instructions vs. textual instructions vs. an equal number of random tokens
no tuning vs. gradient tuning of the instructions part of the prompt (if any)
no examples vs. some of the examples vs. all of the examples vs. an equal number of random tokens
no tuning vs. gradient tuning of the examples part of the prompt (if any)
whether gradient tuning of the prompt tunes the layer-0 embeddings or the KV cache
whether gradient tuning of the prompt tunes all tokens, or just the last one as is common for steering (see section 6.1.2 of this survey)
no tuning vs. gradient tuning of the model parameters (e.g., PEFT); could also consider TTT
This would then allow separating out the effects of the different choices, possibly by fitting a simple predictive model (e.g., linear regression) to all the results.

Question from AC to Reviewer wD3o
I'm having trouble understanding your point here. They are trying to solve a few-shot learning problem, right? In other words, learn an input → output mapping given a small training set. Here are some possible ways to do it:

Method 1: Don't use any prompt. Just adapt the LLM's parameters on the small training set examples using any generic PEFT method. They did this in their initial rebuttal (I think?), because they thought that's what you wanted.

Method 2: Don't adapt any parameters. Just prompt with the small training set and rely on ICL to continue the pattern on a test input. This is their baseline.

Method 3: Like method 2, but additionally adapt the prompt encoding so that it performs well under leave-one-out. This is their main method. It trains a relatively small number of parameters, like method 1, but not the Transformer parameters. It's an application of Prompt Tuning to ICL prompts rather than instruction prompts.

Are we in agreement about the above? If so, what did you want them to do differently?

Reply from Reviewer wD3o
Your understanding is correct, while one thing worth noticing is that any adaptive parameter in Method 1 (PEFT) can be adaptive in Method 3 (ICO) and vice versa, including transformer parameters (e.g., LORA) and caches (prefix/prompt tuning). The inherent difference is about if it is adaptive to a dataset or a input.

My main concern left now is how small the training set is (the complete training set from which few-shot examples of each input are chosen), to make the proposed ICO beneficial.

If the training set is big enough, I would certainly prefer Method 1, maybe with larger adaptive parameters, as it can utilize the complete training set.

If the training set is small enough, the proposed Method 3 also show little advantage over M1 and M2. As mentioned in above parts and reviews, the proposed method is likely to be very sensitive to training set size and ICL example scale and selection. These concerns have not been appropriately discussed yet. Thus I give my negative rating.
