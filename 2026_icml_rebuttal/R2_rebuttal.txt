## (W1 + Q1) Base Model Generalization for ARC.

We clarify that fine-tuning on the ARC training split is standard practice: both TTT and "Searching Latent Program Spaces" (Section 2) adopt the same approach. The fine-tuning teaches the model to parse the grid input-output format, not to solve specific test tasks (training and test tasks are entirely disjoint). Without it, Llama3.2-1B scores 0% on ICL. All methods in our ARC comparisons use the same fine-tuned base model, ensuring a fair evaluation.

That said, we agree that demonstrating CT-KV on a fully off-the-shelf model strengthens the paper. We evaluate on ARC using Qwen3-8B with **no task-specific fine-tuning**, lowering LR to 1e-3 and performing 50 steps for each of 400 ARC tasks.

|Method|ARC Acc. (%)|
|-|-|
|Prefix Tuning|0.0%|
|ICL|3.5%|
|CT-KV (no LOO)|4.0%|

CT-KV with LOO achieves 4.8%, a 37% relative improvement over ICL's 3.5%. Prefix Tuning scores 0% because it optimizes randomly initialized vectors, failing to leverage the model's spatial reasoning priors. This demonstrates CT-KV generalizes to fully unseen task formats.

**LOO on off-the-shelf models.** Interestingly, LOO masking helps on Qwen3-8B (4.8% vs 4.0%), contrary to Section 5.7 where it hurts on the fine-tuned Llama3.2-1B. This is encouraging: CT-KV with LOO is broadly effective on off-the-shelf models, and its past failure on ARC appears specific to the fine-tuned model rather than a fundamental limitation.

## (Q3) Serving Scalability.

We thank the reviewer for recognizing CT-KV's training efficiency.

**Intended use case.** Context Tuning is designed as an inference-time optimization method where the KV cache is optimized on-the-fly for a given query and discarded afterward, so persistent storage and swapping are not the intended use case.

**Storage and swapping.** If one were to store optimized prefixes, CT-KV's memory footprint/overhead is modest: the same size as ICL's KV cache from the same demonstrations, with fewer parameters than LoRA at the demonstration counts we use (Appendix F).

**Inference efficiency.** When loaded from storage, inference is similar to ICL: prepend the cache and generate. CT-KV's cache is already precomputed, whereas ICL must prefill all demonstration tokens through the model at each inference call. Skipping this prefill for the typically long demonstration context gives CT-KV a significant latency advantage over ICL at serve time.
