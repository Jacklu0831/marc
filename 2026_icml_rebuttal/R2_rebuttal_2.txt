## (W2 + Q2 follow-up) Practical Guidance on Leave-One-Out Masking (LOO).

Regarding the reviewer's concern on whether LOO should be enabled in extreme few-shot settings, the user guidance is simply: **always enable LOO for any off-the-shelf model and any task**.

We summarize the evidence from our previous response: we compared CT-KV with and without LOO, where CT-KV with LOO always outperforms without LOO at extremely low demonstration counts ($k$=2,3,4,6). ## (Q3 follow-up) Serving Scalability.

If ICL stores the KV cache computed from its demonstration tokens (as many serving systems do), CT-KV would store a KV prefix identical in size to ICL's, meaning storage, swapping, and inference time are **exactly identical**. However, the reviewer assumes that ICL stores raw text tokens instead; **the yes-or-no answer in this case is that CT-KV has higher storage and lower latency**, as quantified below. All results are averaged across 10 runs on BBH and MMLU, run on an L40S GPU.

**Storage.**

|Method|BBH|MMLU|
|-|-|-|
|ICL (text tokens)|4 KB|3 KB|
|CT-KV (KV prefix)|28 MB|21 MB|

For reference, when the same demonstrations are reused across queries, storing only text tokens is less common in LLM deployment due to the inference latency cost (quantified below). Additionally, TTT's LoRA adapter (rank 64) requires 92 MB regardless of task, more than both ICL and CT-KV. Therefore, despite CT-KV's KV prefix being larger than raw text tokens, it is small compared to existing adapters like LoRA. For even a thousand user tasks, the total storage of CT-KV would only be 28 GB.

**Inference latency.** LLM generation has a *prefill* phase (process input tokens, produce first output token) and a *decoding* phase (generate subsequent tokens). ICL prefills all demonstration tokens and the query. CT-KV swaps in a precomputed KV prefix and prefills only the query. After prefill, both have the same KV cache size, so decoding cost is identical. The only difference is the time-to-first-token (TTFT), which we measure below in two scenarios for CT-KV's prefix storage: on disk (slowest) and in CPU memory (typical for serving systems that keep hot data in RAM).

|Method|BBH|MMLU|
|-|-|-|
|ICL|17.55 ms|13.75 ms|
|CT-KV (swapped from disk)|15.72 ms|11.34 ms|
|CT-KV (swapped from CPU)|**11.07 ms**|**9.96 ms**|

CT-KV is significantly faster in all settings. The speedup comes from skipping the demonstration prefill, so it grows with more demonstrations (longer prefill length) and with larger models (more computation saved per token).
